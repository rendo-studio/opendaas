# OpenDaaS 文档站运行时规范
## OpenDaaS Site Runtime Specification v1

**版本：** v1.0  
**状态：** 当前主稿  
**产品工作名：** `OpenDaaS`  
**CLI 名：** `opendaas`  
**性质：** 实施规范  
**依赖：** [CLI 命令面规范](./CLI-SPEC.md)、[文档站信息架构与渲染规范](./DOCS-SITE-SPEC.md)、[实时文档站与编辑边界规范](./LIVE-DOCS-SPEC.md)、[命名空间规范](./NAMESPACE-SPEC.md)

---

## 1. 文档目的

本文档定义 OpenDaaS 如何把一个现有文档包渲染为人类可访问的文档站。

它要回答的是：

- 输入到底是项目根路径还是文档包路径
- 文档站构建产物应落在哪里
- CLI 应如何直接返回可访问链接
- 为什么站点产物不应写入项目仓库
- 本地链接与未来远程发布的边界是什么

---

## 2. 基本判断

OpenDaaS 文档站的默认形态应是：

1. 从指定文档包路径即时渲染
2. 产物写入全局运行时目录，而不是项目仓库
3. CLI 直接返回本地可访问链接
4. 文档站仍然只是 `docs/` 的派生层，不是独立真相源

该默认形态服务的是项目内部协作可视化，不是项目或框架的对外发布站。

一句话说：

> **OpenDaaS 默认采用“本地隐式部署”模式：CLI 读取文档包、在全局目录构建派生站点、启动本地服务并返回访问链接。**

---

## 3. 输入路径模型

`opendaas site` 至少应支持两类输入：

### 3.1 项目根路径

当输入为项目根路径时，CLI 应自行解析：

- `docs/` 共享文档包
- `.opendaas/` 中与站点渲染有关的最小控制面状态

### 3.2 文档包路径

当输入直接指向一个文档包路径时，CLI 应把该路径视为渲染源。

这意味着：

- 即使没有完整仓库，也可以从一个独立文档包启动站点
- 只要文档包本身完整，CLI 就应能生成可访问站点

---

## 4. 真相源边界

在站点运行时阶段：

1. `docs/` 是唯一正式渲染源
2. `.opendaas/` 只为 CLI 提供必要控制面状态
3. 文档站不应反向成为真相源
4. 人类可见的高层状态必须先存在于 `docs/`

这意味着：

- 不能把只有 `.opendaas/` 才知道的信息直接偷偷渲染进站点
- 如果某项信息需要让人类看到，就应先提升到 `docs/`

---

## 5. 为什么不写入项目仓库

站点构建产物默认不应写入项目仓库。

原因是：

1. 站点产物本质上是派生文件，不是项目真相源
2. 写回仓库会污染项目目录并制造无意义的工作区噪声
3. 不同文档站主题或构建器切换时，派生产物不应进入版本管理
4. 用户需要的是“直接看站点”，不是“多出一堆构建目录”

因此，当前正式结论是：

> **OpenDaaS 文档站产物应默认落在 CLI 自己的全局运行时目录，而不是项目仓库内部。**

当前实现备注：

- 当前站点模板属于 OpenDaaS 包自身的 `site-runtime/` 模板目录
- 实际运行时实例目录已经移动到系统全局 runtime 目录
- 每个 docs/workspace 源会得到独立的 `site-id` 与 runtime root

---

## 6. 全局运行时目录

OpenDaaS 应维护一个对用户默认隐式的全局运行时目录。

它至少应承担：

1. 站点构建产物缓存
2. 站点实例索引
3. 本地运行时元信息
4. staged docs 内容
5. runtime-data 快照

逻辑上建议至少包含：

```text
<global-runtime>/
  sites/
    <site-id>/
      content/
      runtime-data/
      .next/
  registry/
    sites.json
```

说明：

- `<global-runtime>` 应位于系统应用数据目录，而不是项目仓库
- `<site-id>` 可由项目路径或文档包路径哈希得出
- `content/` 存放 staged docs
- `runtime-data/` 存放控制面快照、版本戳与站点元信息
- `.next/` 存放本地 dev/build 产物

---

## 7. 本地访问模式

### 7.1 默认本地访问

CLI 默认应启动一个仅面向本机的本地站点服务，并返回可访问链接。

返回结果至少应包含：

1. 本地 URL
2. 源路径
3. 站点实例标识

### 7.2 默认回环地址

默认服务地址应为本地回环地址，例如：

```text
http://127.0.0.1:<port>
```

默认不应直接暴露为公网地址。

### 7.3 路径变化后的重建

当 `docs/` 发生变化时：

- `site open` 模式应支持 watch 与自动重建
- `site open` 启动瞬间应保证当前站点是最新快照

---

## 8. CLI 命令职责建议

### 8.1 `opendaas site open [path]`

职责：

1. 解析项目根路径或文档包路径
2. 生成或复用全局运行时中的站点产物
3. 启动本地服务
4. 直接返回可访问链接

适用场景：

- 项目协作者已经有文档包，只想立刻打开内部协作文档站

### 8.2 `opendaas site build [path]`

职责：

1. 将站点产物构建到全局运行时目录
2. 不写回项目仓库
3. 返回站点产物位置或实例信息

### 8.3 `opendaas site clean`

职责：

1. 清理全局运行时中的站点缓存与过期实例
2. 不影响源文档包

---

## 9. 与项目仓库的关系

`opendaas site` 默认不应：

1. 在项目仓库中生成 `site/`、`dist/`、`build/` 一类目录
2. 把派生产物提交回项目真相源
3. 让仓库承担运行时缓存职责

如果未来需要导出可部署静态产物，也应由 CLI 显式导出，而不是默认写进仓库。

---

## 10. 远程发布边界

当前版本应把“本地可访问链接”作为默认目标。

未来可以扩展远程发布，但这不属于当前默认运行时模型。

即使未来支持远程发布，它也不应改变默认语义：

- 默认发布对象仍是项目内部协作文档包
- 不默认等同于产品对外文档站
- 不默认承担框架自身的对外教程职责

原因是：

1. 远程链接意味着额外托管面
2. 它会引入鉴权、存储、失效与共享边界问题
3. 这不应阻塞 CLI-first 本地体验

因此，当前正式结论是：

> **OpenDaaS 先做本地隐式部署与本地可访问链接；远程发布是后续增强，不是当前前提。**

---

## 11. 不合格状态

出现以下任一情况时，应视为文档站运行时规范未正确落地：

1. 站点构建产物默认写回项目仓库
2. CLI 无法从独立文档包路径启动站点
3. CLI 启动站点后不能直接返回可访问链接
4. 站点依赖 `.opendaas/` 私有状态补全人类可见现实
5. 站点默认暴露为公网而不是本地回环访问

---

## 12. 当前结论

当前可以明确：

> **OpenDaaS 文档站应默认采用本地隐式部署模式。**

更具体地说：

> **CLI 接收项目根路径或文档包路径作为输入，在全局运行时目录生成站点产物，启动本地服务并直接返回可访问链接；站点产物默认不写回项目仓库。**
