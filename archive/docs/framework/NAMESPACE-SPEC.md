# OpenDaaS 命名空间与可见性规范
## OpenDaaS Namespace Specification v1

**版本：** v1.0  
**状态：** 当前主稿  
**产品工作名：** `OpenDaaS`  
**CLI 名：** `opendaas`  
**性质：** 核心规范  
**依赖：** [PRD](./PRD.md)、[CLI 命令面规范](./CLI-SPEC.md)、[项目文档包最小组成规范](./PROJECT-PACK-SPEC.md)、[概念定义](./CONCEPTS.md)

---

## 1. 文档目的

本文档用于定义 `OpenDaaS` 中三个核心命名空间的边界：

1. `docs/`
2. `.opendaas/`
3. 文档站派生命名空间

它要回答的是：

- 哪些内容属于共享项目现实
- 哪些内容只属于内部控制面
- 哪些内容可以渲染给人类
- 哪些内容不应暴露到文档站
- 哪些内部信息必须被提升到 `docs/`

---

## 2. 基本判断

在 `OpenDaaS` 中：

1. `docs/` 是共享项目文档包
2. `.opendaas/` 是内部控制面工作区
3. 文档站只是从 `docs/` 派生的人类友好展示层

这意味着：

- `docs/` 是人类与 Agent 共用的主真相源
- `.opendaas/` 不是项目文档包本体
- 文档站不是根真相源
- 文档站默认是项目内部协作文档站，而不是产品对外文档站

---

## 3. `docs/` 的定位

`docs/` 的本质不是“只给人类看的说明文档”，而是共享项目现实。

它默认服务项目内部协作，而不是产品外部受众。

它应同时满足：

1. 人类可读
2. Agent 可读
3. 可版本管理
4. 可迁移
5. 可被渲染为常规技术文档站

同时，`docs/` 的结构不应被完全写死。

更合理的要求是：

- 保留少量固定锚点
- 其余内容根据项目类型自适应扩展

在结构气质上，`docs/` 应更接近成熟项目的技术文档站，例如：

- 项目介绍
- 功能或模块说明
- 协议与契约
- API
- 开发说明
- 部署说明

同时，OpenDaaS 还要求它额外承载：

- 当前状态
- 当前 active change
- 高层开发计划
- 面向人类可审视的风险与验证结果

一句话说：

> **`docs/` 是“常规技术文档站结构 + 项目级协作推进层”。**

---

## 4. `.opendaas/` 的定位

`.opendaas/` 是 OpenDaaS 的内部控制面工作区。

它不是纯临时目录，也不是项目文档包本体。

与 `docs/` 不同，`.opendaas/` 更适合采用固定结构，以保证 CLI 与不同 Agent 环境能稳定读写。

它主要承载：

1. 框架接入标识
2. CLI 内部配置
3. 差异索引与差异确认状态
4. Agent 细粒度开发计划
5. 函数级、组件级、局部实现级 TODO
6. 仅服务当前开发轮次的中间工件

这些内容对控制流程很重要，但默认不应直接暴露给人类侧文档站。

---

## 5. 文档站派生命名空间的定位

文档站是 `docs/` 的派生展示层。

它默认面向：

- 项目开发者
- 项目维护者
- 开发端 Agent 的协作审视

它默认不承担：

- 产品官网
- 用户帮助中心
- 消费端 Agent 文档面

它应满足：

1. 只从 `docs/` 源文档派生
2. 不直接依赖 `.opendaas/` 作为公开展示输入
3. 不反向成为项目的唯一真相源

如果文档站支持编辑，则编辑结果必须回写到 `docs/` 源文档，而不是只停留在站点数据库或页面状态中。

---

## 6. 可见性规则

### 6.1 默认可见于文档站的内容

通常包括：

- 项目介绍
- 模块与功能说明
- API 与协议
- 开发与部署说明
- 高层计划
- 当前状态
- 当前 active change
- 面向人类的重要风险与验证结果

### 6.2 默认不暴露到文档站的内容

通常包括：

- `.opendaas/` 内部配置
- 差异缓存或差异确认状态
- Agent 的函数级、组件级细粒度工作计划
- 当前开发轮次的微观 TODO
- 仅服务 CLI 的中间文件

---

## 7. 提升规则

并不是所有 `.opendaas/` 中的内容都应永久留在内部。

只要某项内部信息已经影响到项目级理解，就必须被提升到 `docs/`。

至少以下情况必须提升：

1. 改变了项目目标或范围
2. 改变了当前高层计划
3. 改变了 active change 的范围、状态或优先级
4. 暴露了新的项目级风险
5. 形成了新的协议、契约、API 或架构边界
6. 影响人类接手、审视或决策判断

一句话说：

> **微观实现细节可以留在 `.opendaas/`，项目级现实必须进入 `docs/`。**

---

## 8. 差异追踪规则

### 8.1 差异追踪对象

`opendaas` 的差异追踪必须以 `docs/` 的共享源文档为核心对象。

### 8.2 差异来源

差异来源至少包括：

1. 人类在 IDE 中修改 `docs/`
2. 人类通过文档站修改并回写 `docs/`
3. 开发端 Agent 修改 `docs/`

### 8.3 差异查询要求

CLI 必须支持：

1. 行级差异查询
2. 当前未确认差异查询
3. 差异确认或基线更新

### 8.4 任务前检查要求

开发端 Agent 在每轮任务开始前，必须先通过 CLI 检查 `docs/` 的未确认差异。

如果差异已经改变当前理解，Agent 必须先重新吸收这些变化，再开始本轮工作。

---

## 9. 与目录协议的关系

本规范只定义命名空间与可见性边界，不定义最终具体路径。

后续目录协议至少必须回答：

1. `docs/` 下各逻辑分区如何映射到具体目录和文件
2. `.opendaas/` 下哪些内部工件默认存在
3. 哪些工件允许派生到站点，哪些不允许
4. 差异索引与差异确认状态的具体落点是什么

---

## 10. 当前结论

当前可以明确：

> **`docs/` 与 `.opendaas/` 不是同一个东西。**

更具体地说：

> **`docs/` 承担共享项目现实，`.opendaas/` 承担内部控制面工作区；文档站只应从 `docs/` 派生，任何已经影响项目级理解的内部信息都必须被提升到 `docs/`。**
