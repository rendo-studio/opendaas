# OpenDaaS 共享项目文档包规范
## OpenDaaS Shared Docs Package Specification v2

**版本：** v2.1  
**状态：** 当前主稿  
**产品工作名：** `OpenDaaS`  
**CLI 名：** `opendaas`  
**性质：** 核心规范  
**依赖：** [PRD](./PRD.md)、[CLI 命令面规范](./CLI-SPEC.md)、[概念定义](./CONCEPTS.md)、[命名空间规范](./NAMESPACE-SPEC.md)

---

## 1. 文档目的

本文档只定义 `docs/` 共享项目文档包。

它不定义 `.opendaas/` 内部工作区，也不定义最终目录协议的全部路径细节。

它要回答的是：

- `docs/` 应该成为什么样的文档包
- 哪些内容必须进入 `docs/`
- 哪些内容不应进入 `docs/`
- `docs/` 最少应具备哪些固定锚点
- `docs/` 哪些部分应保持自适应扩展

一句话说：

> **本规范定义的是“对人类与 Agent 共用的共享项目文档包”，而不是 OpenDaaS 的内部流程工作区。**

---

## 2. 基本判断

在 `OpenDaaS` 中，`docs/` 的定位必须非常明确：

1. 它是共享项目现实
2. 它是人类与 Agent 共用的主真相源
3. 它应看起来像一个常规技术文档站，而不是内部流程垃圾场
4. 它需要支持迁移、接手、继续迭代和必要时的项目重建
5. 它不负责承载所有内部开发细节

这里的“常规技术文档站”默认指项目内部协作和开发使用的技术文档站，而不是产品官网、用户帮助中心或消费端 Agent 文档面。

这意味着：

- `docs/` 不是只写给人类看的“展示材料”
- `docs/` 也不是只写给 Agent 看的“spec 文件夹”
- `docs/` 应同时服务于人类理解和 Agent 继续工作
- `docs/` 还将作为后续 OpenDaaS skill 包的权威来源之一

---

## 3. 本规范不做什么

本规范暂不定义：

1. `.opendaas/` 的固定结构
2. `docs/` 的最终具体文件路径
3. 文档站的最终导航与渲染实现
4. 各页面的最终模板字段

这些内容由后续专门规范细化。

---

## 4. 设计原则

### 4.1 常规技术文档站优先

`docs/` 的外观和组织气质，应尽量接近成熟项目的常规技术文档站，而不是暴露内部流程对象。

这里追求的是内部协作可读性，不是对外营销或用户教育页面。

### 4.2 人类与 Agent 共享同一视角

Agent 应以与人类相同的 `docs/` 共享文档包作为主上下文，而不是单独维护一套平行的 Agent 专用项目现实。

### 4.3 项目级信息进入 `docs/`

凡是影响项目级理解、项目级约束、项目级范围、项目级风险、项目级协作判断的信息，都必须进入 `docs/`。

### 4.4 微观实现细节不默认进入 `docs/`

函数级、组件级、局部实现级、当前开发轮次的细粒度内部计划，不应默认塞进 `docs/`。

### 4.5 固定锚点 + 自适应扩展

`docs/` 不应完全自由散乱，也不应被写死成对所有项目都一样的僵硬结构。

更合理的方式是：

- 固定少量必须存在的共享锚点
- 其余结构根据项目类型自适应扩展

### 4.6 文档站只是 `docs/` 的派生形态

文档站应从 `docs/` 派生，不能反过来成为共享项目现实的唯一来源。

---

## 5. `docs/` 的固定锚点

不管项目是什么类型，`docs/` 至少应提供以下固定锚点。

这里的“锚点”是逻辑能力要求，不强制等于某个固定路径名。

### 5.1 项目入口锚点

必须回答：

- 项目是什么
- 项目为什么存在
- 当前总体目标是什么
- 对新接手的人或 Agent，应从哪里开始读

### 5.2 最终目标锚点

必须回答：

- 最终目标是什么
- 为什么这是当前最终目标
- 完成标准是什么
- 当前明确不做什么

### 5.3 当前状态锚点

必须回答：

- 项目当前推进到哪里
- 当前阶段是什么
- 当前高层风险或 blocker 是什么
- 下一步主要动作是什么
- 当前进度大致到哪里

### 5.4 当前工作锚点

必须回答：

- 当前 active work 是什么
- 当前高层 change / feature / page / capability 焦点是什么
- 当前高层计划是什么

### 5.5 开发入口锚点

必须回答：

- 如何本地运行
- 如何开发
- 如何测试
- 如何构建
- 如何部署或验证

### 5.6 变化可见锚点

必须回答：

- 最近发生了什么重要变化
- 当前 change 的高层范围和结果是什么
- 人类应该重点关注哪些变化

---

## 6. `docs/` 的自适应扩展区

除了固定锚点之外，`docs/` 的主体结构应根据项目类型自适应扩展。

常见的自适应分区包括，但不限于：

1. 项目介绍
2. 功能或模块说明
3. 协议与契约
4. API 参考
5. Guides / Tutorials
6. Architecture / Engineering
7. Deployment / Operations
8. Integrations
9. Releases / Changelog

并不是每个项目都必须同时拥有这些区块。

原则是：

- 有实际内容需求才建
- 没有必要就不要为了凑结构而硬建空目录

---

## 7. 何种内容必须进入 `docs/`

以下内容应视为共享项目现实，必须进入 `docs/`：

1. 项目介绍与目标
2. 当前范围与关键边界
3. 高层功能或模块说明
4. 对外或对内的重要协议、契约、API
5. 当前高层开发计划
6. 当前状态与 active work
7. 重要风险与验证结果
8. 工程开发、运行、测试、部署说明
9. 新 Agent 或新人类维护者接手所需的核心上下文
10. 标准头部元信息

---

## 8. 何种内容不应默认进入 `docs/`

以下内容默认不应进入 `docs/`，除非它们已经上升为项目级理解的一部分：

1. 函数级实现计划
2. 组件级微观任务拆分
3. 仅服务当前开发轮次的局部 TODO
4. 差异缓存与差异确认状态
5. CLI 内部配置
6. Agent 自用的中间笔记
7. 纯内部流程控制信息

这些内容更适合存在于 `.opendaas/`。

---

## 9. 文档元信息要求

`docs/` 中的每个文档都必须具有标准头部元信息。

当前第一版最少字段为：

1. `name`
2. `description`

这项要求既服务文档站索引，也服务后续 skill 打包。

---

## 10. 人类可读性要求

`docs/` 虽然同时服务 Agent，但它的表达仍应优先保持人类可读。

这意味着：

1. 页面主题应围绕项目理解，而不是围绕内部流程对象命名
2. 内容应尽量使用人类熟悉的技术文档表达方式
3. 不应把大量内部细粒度任务、差异状态、工作缓存直接暴露给人类

换句话说：

> **人类看到的 `docs/`，应该更像 Vue、Next.js 一类成熟项目的技术文档站，而不是一堆 AI 工作台转储文件。**

---

## 11. Agent 可用性要求

`docs/` 不是只为了好看，它必须足以支撑 Agent 在新环境中恢复项目上下文。

因此，`docs/` 至少应让 Agent 快速回答：

1. 项目是什么
2. 当前目标和范围是什么
3. 现在正在做什么
4. 当前高层计划是什么
5. 当前主要变化和风险是什么
6. 如何运行、测试、部署
7. 哪些模块、协议、契约或 API 已经存在
8. 当前最终目标是什么
9. 当前进度大致到哪里

如果 Agent 仍然必须依赖旧聊天才能回答这些问题，那么 `docs/` 还不合格。

---

## 12. 最低合格条件

一个合格的 `docs/` 共享项目文档包，至少应具备：

1. 项目入口锚点
2. 最终目标锚点
3. 当前状态锚点
4. 当前工作锚点
5. 开发入口锚点
6. 变化可见锚点
7. 至少一组适合当前项目的自适应内容区块

这七项一起成立，`docs/` 才算不是空壳。

---

## 13. 不合格状态

出现以下任一情况时，应视为 `docs/` 仍不合格：

1. 只有零散 Markdown，没有清晰入口
2. 没有最终目标锚点
3. 人类无法快速知道当前在做什么
4. Agent 无法仅凭 `docs/` 恢复高层项目上下文
5. 文档里只有内部流程痕迹，没有正常技术文档结构
6. 高层变更和重要风险没有对人可见的落点
7. 工程接手所需的开发、运行、测试、部署信息缺失
8. 文档缺少标准头部元信息

---

## 14. 与 `.opendaas/` 的关系

`docs/` 与 `.opendaas/` 的关系必须满足：

1. `docs/` 承担共享项目现实
2. `.opendaas/` 承担内部控制面工作区
3. 微观实现细节默认留在 `.opendaas/`
4. 任何上升为项目级理解的信息，都必须从 `.opendaas/` 提升到 `docs/`

---

## 15. 与 CLI 的关系

`opendaas` 至少应围绕 `docs/` 提供以下能力：

1. 初始化共享文档包的固定锚点
2. 为不同项目类型提供示范模板
3. 校验固定锚点是否缺失
4. 对共享源文档提供差异查询
5. 从原生结构化 goal / plan / task / progress 同步高层摘要到 `docs/`
6. 从 `docs/` 渲染文档站

这意味着，CLI 应帮助生成“像正常技术文档站”的起始骨架，而不是生成一套只对 OpenDaaS 自己友好的内部目录。

---

## 16. 与 skill 打包的关系

`docs/` 不只是文档站素材，还应作为后续 OpenDaaS skill 包的权威来源之一。

因此：

1. 文档结构必须稳定
2. 元信息必须标准化
3. 文档类型必须可识别

对 OpenDaaS 框架本身而言，默认使用指南更适合通过 CLI 和 skill 暴露，而不是要求共享项目文档包同时承担框架产品教程。

---

## 17. 当前结论

当前可以明确：

> **`docs/` 应该被设计成共享项目文档包，而不是 `.opendaas/` 的公开镜像。**

更具体地说：

> **`docs/` 必须保留少量固定锚点，其中最终目标锚点是最高优先级之一；整体气质应接近常规技术文档站；`.opendaas/` 内部细粒度计划、差异状态和流程控制信息，不应默认暴露给人类。**
